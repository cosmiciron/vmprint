# TOC VPX - Property Design

This document surveys how tables of contents work across common document domains, then derives a property schema for the TOC VPX template tag. The goal is a system that is immediately useful at its simplest and fully expressive at its most detailed.

This is a TOC-only surface. Adjacent frontmatter artifacts such as List of Figures, List of Tables, and List of Equations should be designed as separate supplementary components rather than folded into the TOC VPX.

---

## 1. Domain Survey

Before defining properties, it is worth observing what a TOC actually does in each major document context. The differences drive every design decision.

### 1.1 Novel / Literary Fiction

The TOC is a navigational aid, not a structural map. Readers want chapter names, not an outline. The aesthetic matters: it should feel like part of the book.

Typical characteristics:
- Depth 1 or 2, chapters and optionally parts above them
- No heading numbers
- Titles may be stylized, for example italics or small caps
- Leaders: dots or flush-right with whitespace gap
- Page numbers: yes, but subordinate to the title
- Part headings may appear as dividers rather than entries
- No decorative numbering generated by the TOC

### 1.2 Academic Paper

Short journal articles often omit a TOC. Longer papers and monographs include one, and it closely mirrors the document's numbered section structure.

Typical characteristics:
- Depth 2-3
- Section numbers shown, inherited from heading numbering
- Leaders: dots
- Page numbers: yes
- Entries are compact; visual hierarchy comes from indentation and weight

### 1.3 Thesis / Dissertation

Institutional documents with strict formatting requirements. The TOC is one of several formal frontmatter lists.

Typical characteristics:
- Depth 3-4
- Full outline numbering
- Front matter may use roman numerals while body uses arabic numerals
- Major headings sometimes in caps or small caps
- Strict indentation rules, often specified by the institution

### 1.4 Technical Manual / Documentation

The TOC is a primary navigation tool, often the first thing a reader uses. Depth and discoverability matter more than elegance.

Typical characteristics:
- Depth 3-5
- Numbered entries
- Chapter or section prefix on top-level entries, for example `Chapter 3`
- Leaders: dots
- Page numbers: yes
- May have a separate brief contents in some publishing programs

### 1.5 Legal Document

Legal structure uses a defined hierarchy, often Article / Section / Subsection, with its own labeling convention. Labels come from document semantics, not from a generic TOC counter.

Typical characteristics:
- Depth 2-3
- Entry labels carry semantic weight, for example `Article I`, `Section 2.3`, `Section 4(b)`
- Leaders: dots or none
- Page numbers: yes, sometimes formatted as `p. 12`
- Frontmatter may appear under distinct labels

### 1.6 Business Report / White Paper

Clean, professional, and brief. The TOC is there to show structure and aid navigation, not to exhaustively list every subsection.

Typical characteristics:
- Depth 1-2
- Section numbers optional
- Leaders: dots or a thin rule
- Page numbers: yes
- Titles may use brand typography

### 1.7 Magazine / Editorial

The TOC is a feature list, not an outline. It names articles, features, or departments with page numbers. Visual design dominates.

Typical characteristics:
- Depth 1, often flat
- No section numbering
- Page reference style varies, for example `42` or `p. 42`
- No leaders, or a decorative separator
- Often multi-column
- Entry format may be richer than title plus page

## 2. Design Principles

From the survey, three tensions are clear.

**Depth vs. brevity.** Most documents want either a shallow summary TOC or a full hierarchical one. Both should be easy to configure.

**Inherited labels vs. generated labels.** Technical and academic documents often display the same numbering that appears in the headings. Legal documents use semantic labels. Novels use no labels at all. The property system must accommodate all three without forcing them into one overloaded switch.

**Uniform entries vs. per-level control.** Simple documents need uniform properties across all levels. Complex documents need distinct treatment per level. The property system should reward simplicity with brevity and unlock per-level control when needed.

The solution is a graduated schema:

- top-level shorthand properties handle the common case
- a `levels` array provides full per-level overrides when needed
- a `variant` preset covers common domain starting points

## 3. Property Reference

### 3.1 `variant`

```ts
variant?: string
```

A named preset that establishes sensible defaults for a document domain. Any explicitly provided property overrides the preset.

| Value | Description |
|---|---|
| `"novel"` | Depth 2, no labels, dot leaders, page numbers |
| `"academic"` | Depth 3, inherited labels, dot leaders, page numbers |
| `"thesis"` | Depth 4, inherited labels, dot leaders, page numbers |
| `"technical"` | Depth 4, inherited labels, dot leaders, page numbers, chapter prefix on level 1 |
| `"legal"` | Depth 3, inherited semantic labels, dot leaders, page numbers with `p. ` prefix |
| `"report"` | Depth 2, no labels, dot leaders, page numbers |
| `"editorial"` | Depth 1, no labels, no leaders, bare page numbers |

Variants are defaults, not constraints.

---

### 3.2 `title`

```ts
title?: string | false
titleRole?: string
```

`title` is the heading text rendered above the TOC entries. Defaults vary by variant, for example `Contents` or `Table of Contents`. Set it to `false` to suppress the title entirely.

`titleRole` is the style role applied to the title element. Defaults to `toc-title`.

---

### 3.3 `depth`

```ts
depth?: number  // default: 3
```

The maximum heading depth to include. Headings deeper than this are omitted.

---

### 3.4 `include` and `exclude`

```ts
include?: string[]
exclude?: string[]
```

By default, the TOC includes all headings up to `depth`. Use `include` to restrict to specific semantic roles, or `exclude` to suppress specific ones.

Examples:

```json
{ "include": ["chapter", "section"] }
{ "exclude": ["h4", "h5"] }
{ "include": ["part", "chapter"] }
```

Role names match the role strings used in the document's heading elements.

---

### 3.5 `scope`

```ts
scope?: "document" | "section" | string  // default: "document"
```

Controls which headings are collected.

| Value | Behavior |
|---|---|
| `"document"` | All headings in the document |
| `"section"` | Only headings within the same local section as the placeholder |
| `string` | A named scope selected by host-defined conventions |

This property is intentionally provisional. The final engine-facing boundary model for local TOCs still needs a more formal definition.

---

### 3.6 `leaders`

```ts
leaders?: "dots" | "dashes" | "line" | "none" | LeaderConfig
```

The fill between the entry title and the page number.

| Value | Appearance |
|---|---|
| `"dots"` | `Title ............. 42` |
| `"dashes"` | `Title - - - - - - - 42` |
| `"line"` | `Title __________ 42` |
| `"none"` | `Title              42` |

```ts
interface LeaderConfig {
  char: string
  spacing?: "tight" | "normal" | "loose"
}
```

---

### 3.7 `pageNumbers`

```ts
pageNumbers?: boolean | PageNumberConfig  // default: true
```

Controls how page numbers appear on entries. `false` suppresses them entirely. `true` uses the default format.

```ts
interface PageNumberConfig {
  show?: boolean
  format?: "arabic" | "roman-lower" | "roman-upper" | "alpha-lower" | "inherit"
  prefix?: string
  suffix?: string
}
```

`inherit` uses whatever page number format is active at the referenced page.

---

### 3.8 `labels`

```ts
labels?: false | "inherit" | "generate"
```

Controls whether entries display a leading label before the title.

| Value | Behavior |
|---|---|
| `false` | No label is shown |
| `"inherit"` | Use the heading's own document label or number |
| `"generate"` | Generate TOC-local labels according to `generatedLabels` |

This property replaces the older overloaded notion of TOC `numbering`. It intentionally allows legal and semantic labels to live in the same conceptual slot as numeric labels.

---

### 3.9 `generatedLabels`

```ts
generatedLabels?: "outline" | "decimal"
```

Controls how TOC-local labels are generated when `labels` is set to `"generate"`.

| Value | Behavior |
|---|---|
| `"outline"` | `1`, `1.1`, `1.1.2` |
| `"decimal"` | Flat decimal labels per level |

If `labels` is not `"generate"`, this property has no effect.

---

### 3.10 `indent`

```ts
indent?: boolean | number | IndentConfig  // default: true
```

Controls per-level indentation of entries.

| Value | Behavior |
|---|---|
| `false` | No indentation; all entries flush left |
| `true` | Default per-level indent |
| `number` | Points of indent per level |

```ts
interface IndentConfig {
  perLevel?: number
  levels?: Record<number, number>
}
```

---

### 3.11 `frontmatter`

```ts
frontmatter?: "include" | "exclude" | "separate"  // default: "include"
```

Controls how headings from frontmatter pages are handled.

| Value | Behavior |
|---|---|
| `"include"` | Frontmatter headings appear inline with body headings |
| `"exclude"` | Frontmatter headings are omitted |
| `"separate"` | Frontmatter headings are collected into a distinct section above the main entries |

---

### 3.12 `entryRole`

```ts
entryRole?: string | Record<number, string>
```

The style role applied to TOC entry container elements. A single string applies uniformly. A record maps depth to role.

```json
{ "entryRole": { "1": "toc-part", "2": "toc-chapter", "3": "toc-section" } }
```

Roles must be defined in the document's style table.

---

### 3.13 `parts`

```ts
parts?: EntryParts
```

A TOC entry is a compound element. It has up to four rendered pieces:

- the label
- the title text
- the leader fill
- the page number

Each piece may need independent styling.

```ts
interface EntryParts {
  labelRole?: string
  titleRole?: string
  leaderRole?: string
  pageRole?: string
}
```

`entryRole` is the container role and controls block-level properties such as margin, indentation, or baseline behavior. `parts` controls the inline sub-elements within that container.

---

### 3.14 `spacing`

```ts
spacing?: number | SpacingConfig
```

Controls structural vertical rhythm within the TOC. A plain number sets uniform space between entries at all levels.

```ts
interface SpacingConfig {
  entry?: number
  levelBreak?: number
  afterTitle?: number
  levels?: Record<number, { before?: number; after?: number }>
}
```

This is structural entry spacing owned by the VPX, not typography owned by styles.

---

### 3.15 `pageColumn`

```ts
pageColumn?: "margin" | number  // default: "margin"
```

Controls where the page number is horizontally positioned.

| Value | Behavior |
|---|---|
| `"margin"` | Page number right-aligned to the content area right edge |
| `number` | Fixed tab stop position in points from the left content edge |

This is a structural layout value, not a style property.

---

### 3.16 `prefix`

```ts
prefix?: string | Record<number, string>
```

A text string prepended to entries at the specified depth. A single string applies to all levels. A record maps depth to prefix.

```json
{ "prefix": { "1": "Chapter " } }
```

This is useful for cases such as `Chapter 3` without embedding the prefix in the heading text itself.

---

### 3.17 `levels` - Per-Level Override

```ts
levels?: LevelConfig[]
```

The full-control escape hatch. When present, each entry configures a specific heading depth. Properties specified here take precedence over all top-level shorthand properties for that depth.

```ts
interface LevelConfig {
  depth: number
  include?: boolean
  entryRole?: string
  parts?: EntryParts
  indent?: number
  leaders?: "dots" | "dashes" | "line" | "none" | LeaderConfig
  pageNumbers?: boolean | PageNumberConfig
  labels?: false | "inherit" | "generate"
  generatedLabels?: "outline" | "decimal"
  prefix?: string
  suffix?: string
  spaceBefore?: number
  spaceAfter?: number
}
```

Example:

```json
{
  "levels": [
    {
      "depth": 1,
      "entryRole": "toc-article",
      "indent": 0,
      "leaders": "none",
      "pageNumbers": false
    },
    {
      "depth": 2,
      "entryRole": "toc-section",
      "indent": 24,
      "leaders": "dots",
      "pageNumbers": true,
      "prefix": "§ "
    }
  ]
}
```

## 4. Placeholder Schema

The full template tag shape:

```ts
interface TocProps {
  variant?: string

  title?: string | false
  titleRole?: string

  depth?: number
  include?: string[]
  exclude?: string[]
  scope?: "document" | "section" | string
  frontmatter?: "include" | "exclude" | "separate"

  labels?: false | "inherit" | "generate"
  generatedLabels?: "outline" | "decimal"
  prefix?: string | Record<number, string>

  leaders?: "dots" | "dashes" | "line" | "none" | LeaderConfig
  pageNumbers?: boolean | PageNumberConfig
  pageColumn?: "margin" | number
  indent?: boolean | number | IndentConfig

  entryRole?: string | Record<number, string>
  parts?: EntryParts

  spacing?: number | SpacingConfig

  levels?: LevelConfig[]
}
```

In AST form:

```json
{
  "type": "vpx",
  "name": "TOC",
  "props": { ... }
}
```

## 6. Typography and the Styles Table

The TOC VPX does not carry font, size, or color properties directly. Those belong in the document's `styles` table, keyed by the role names assigned through `entryRole`, `titleRole`, and `parts`.

Every role name used in the TOC props is a reference into `styles`. The VPX wraps each piece of generated content in the appropriate role, and the engine resolves that role through the styles table exactly as it would for any authored element.

Structural layout choices such as `indent`, `spacing`, `leaders`, and `pageColumn` are still part of the TOC surface. They are not typography. They determine how the generated TOC is assembled.

---

## 7. What the TOC VPX Does Not Own

These are out of scope for the props schema.

- Entry typography: handled by the styles table
- Leader and page number visual style: handled by `parts` roles and styles
- Multi-column TOC layout: handled by surrounding AST layout primitives such as `zone-map` or `strip`
- TOC page breaks: controlled by the surrounding layout
- Page number resolution: the engine exposes settled page facts; the VPX reads them
- Supplementary frontmatter lists: List of Figures, List of Tables, List of Equations, and similar artifacts should be designed as separate supplementary components
